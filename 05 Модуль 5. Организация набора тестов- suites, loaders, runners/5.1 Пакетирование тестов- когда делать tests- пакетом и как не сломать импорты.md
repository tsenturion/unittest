# Пакетирование тестов в `unittest`: когда делать `tests/` пакетом и как не сломать импорты и discovery

Вы запускаете `python -m unittest discover` и ожидаете увидеть десятки тестов, а получаете `Ran 0 tests`, либо `ImportError: No module named ...`. В `unittest` это почти всегда означает одно: тесты не «нашлись» не потому, что фреймворк плохо ищет файлы, а потому что он **пытается импортировать модули**, и импорт ломается на структуре проекта или на путях импорта. ([Python documentation][1])

## Discovery в `unittest` — это импорт, а не «поиск по папкам»

`unittest` discovery делает две ключевые вещи:

1. Рекурсивно обходит директории от стартовой точки (`-s/--start-directory`) и выбирает файлы по паттерну (`-p/--pattern`, по умолчанию `test*.py`). ([Python documentation][1])
2. **Преобразует пути в “dotted name” (пунктирные имена модулей) и импортирует их**. Пример из документации: файл `foo/bar/baz.py` будет импортирован как `foo.bar.baz`. ([Python documentation][1])

Это фундаментально: если модуль нельзя импортировать — он не может быть загружен как тестовый модуль, даже если файл физически лежит на диске.

### Схема: «путь → dotted name → import»

```text
tests/unit/test_calc.py
   │
   ├─ discovery нашёл файл по паттерну test*.py
   │
   ├─ превратил путь в dotted name:
   │      tests.unit.test_calc
   │
   └─ попытался сделать import tests.unit.test_calc
          └─ и вот здесь всё ломается, если tests/unit не пакет
```

Эта логика прямо следует из того, что discovery **загружает тесты импортом** и требует, чтобы тестовые файлы были модулями/пакетами, импортируемыми из “top-level directory” проекта. ([Python documentation][1])

## «Импортируемый тестовый модуль» — что требуется от имён и директорий

### 1) Имя файла должно быть валидным идентификатором Python

`unittest` подчёркивает: для совместимости с discovery **имена файлов должны быть валидными идентификаторами**, потому что это модули, которые будут импортироваться. Практический вывод: `test-login.py` (с дефисом) — плохая идея; `test_login.py` — нормальная. ([Python documentation][1])

### 2) Директории на пути dotted name должны быть пакетами (в большинстве реальных случаев)

Если discovery хочет импортировать `tests.unit.test_calc`, то `tests` и `tests/unit` должны быть пакетами, иначе импорт `tests.unit...` невозможен как обычный импорт. В терминах Python **регулярный пакет** — это директория с `__init__.py`. ([Python documentation][2])

Важно: в `unittest` были изменения вокруг namespace packages. В актуальной документации отмечено:

- в Python 3.11 discovery официально **убрал поддержку namespace packages** как стартовой точки; сказано, что стартовая директория и её поддиректории с тестами должны быть **regular package с `__init__.py`**;
- в Python 3.14 discovery снова поддерживает namespace package как start directory, но **тесты не ищутся в поддиректориях без `__init__.py`**, чтобы не сканировать «лишние» директории. ([Python documentation][1])

Практический вывод для учебных/рабочих проектов: если Вы хотите предсказуемое discovery — **ставьте `__init__.py` в `tests/` и во все подкаталоги, где лежат тесты**.

> Короткое правило: discovery импортирует dotted name. Хотите, чтобы dotted name импортировались — обеспечьте регулярные пакеты (`__init__.py`) на пути. ([Python documentation][1])

## Когда делать `tests/` пакетом (то есть добавлять `tests/__init__.py`)

Есть три практических сценария, где `tests` как пакет даёт пользу и снимает ошибки.

### Сценарий A: тесты живут не в корне, а в `tests/` (самый частый)

Если тестовый файл лежит не в корне проекта, discovery почти неизбежно попытается импортировать его как `tests.something...`. Это означает, что `tests/` должен быть пакетом, иначе импорт ломается. ([Python documentation][1])

### Сценарий B: Вы хотите общие хелперы/фикстуры внутри `tests/`

Хелперы вроде `tests/helpers.py`, `tests/factories.py`, `tests/fixtures/...` удобнее импортировать как модули пакета. Python прямо описывает, что пакеты создают иерархию “dotted module names”, а относительные импорты работают только внутри пакета. ([Python documentation][3])

### Сценарий C: Вы хотите запускать подмножество тестов по dotted name

`unittest` CLI позволяет запускать модуль/класс/метод по «квалифицированным именам», а также фильтровать по `-k`, который сопоставляет паттерн с **полным именем теста так, как оно импортировано**. Чем стабильнее dotted name — тем проще точечный запуск. ([Python documentation][1])

## Когда `tests/` может не быть пакетом

Это возможно, но только если структура другая.

Если все тестовые модули лежат **в корне проекта** (например, `test_calc.py` рядом с `myapp/`), dotted name будет просто `test_calc`, и пакет `tests` не нужен. Это рабочая схема для маленьких учебных задач, но в реальных репозиториях быстро превращается в свалку.

## Самый частый источник боли: `sys.path` и “откуда запускаете”

### Что делает `python -m` с путями импорта

При запуске `python -m <module>` интерпретатор ищет модуль по `sys.path` и выполняет его как `__main__`. При этом (по документации) **текущая директория добавляется в начало `sys.path`**, аналогично `-c`. ([Python documentation][4])

Это объясняет, почему «в корне репозитория всё работает», а «из другой директории не работает»: меняется текущая директория → меняется первый элемент `sys.path` → меняется то, что импортируется.

Дополнительно: есть опция `-P`, которая запрещает добавлять потенциально небезопасный путь в `sys.path` (для `-m` — это как раз текущая директория). Если её включить, привычный запуск тестов из корня может перестать видеть Ваш проект. ([Python documentation][4])

### Простейшая проверка, чтобы понимать реальную картину

Сделайте временный файл `debug_path.py` в корне проекта:

```python
# debug_path.py
import sys
from pprint import pprint

print("sys.path[0]:", sys.path[0])
print("first 5 entries:")
pprint(sys.path[:5])
```

Запустите:

```bash
python debug_path.py
python -m debug_path
```

И сравните вывод. Это быстро объясняет, почему импорт «видит» один каталог, а не другой. Механика `sys.path` (первый элемент — директория скрипта или текущая директория при `-m/-c`) описана в документации. ([Python documentation][5])

## Две структуры проекта, которые дают предсказуемые импорты

### Вариант 1: «плоская» структура (проще для старта)

```text
project/
  myapp/
    __init__.py
    calc.py
  tests/
    __init__.py
    unit/
      __init__.py
      test_calc.py
```

Запуск:

```bash
cd project
python -m unittest discover -s tests -t .
```

Почему работает:

- `-t .` задаёт top-level директорию проекта, из которой тесты должны быть импортируемы. ([Python documentation][1])
- `tests` и `tests/unit` — пакеты, значит dotted name `tests.unit.test_calc` импортируется. ([Python documentation][2])

### Вариант 2: `src/`-layout (часто в проде, но требует дисциплины)

```text
project/
  src/
    myapp/
      __init__.py
      calc.py
  tests/
    __init__.py
    test_calc.py
  pyproject.toml
```

Проблема: при обычном запуске `python -m unittest` в `sys.path` попадает `project/`, но **не** `project/src/`, значит `import myapp` может не работать. Это не «ошибка unittest», это следствие того, как формируется `sys.path`. ([Python documentation][5])

Решение “по-взрослому”: установить пакет (часто в editable режиме) и запускать тесты в окружении, где пакет доступен как установленный. В учебной практике можно начать с плоской структуры и перейти к `src/` позже.

## Пример: `tests` как пакет, shared helpers и discovery без сюрпризов

### Код приложения

```python
# myapp/calc.py
def divide(a, b):
    if b == 0:
        raise ZeroDivisionError("b must not be zero")
    return a / b
```

### Хелпер внутри тестов

```python
# tests/helpers.py
def as_float(x):
    return float(x)
```

### Тест

```python
# tests/unit/test_calc.py
import unittest

from myapp.calc import divide
from tests.helpers import as_float  # можно, потому что tests — пакет


class TestDivide(unittest.TestCase):
    def test_divide_ok(self):
        result = divide(10, 4)
        self.assertEqual(as_float(result), 2.5)

    def test_divide_by_zero_raises(self):
        with self.assertRaises(ZeroDivisionError):
            divide(1, 0)
```

Запуск:

```bash
python -m unittest discover -s tests -t .
```

Ключевой момент: импорт `from tests.helpers import ...` работает только если `tests` — импортируемый пакет. Python определяет регулярный пакет через наличие `__init__.py`, и при импорте пакетного модуля `__init__.py` выполняется. Поэтому держите `tests/__init__.py` пустым или максимально нейтральным. ([Python documentation][2])

## Ловушка: `load_tests` в `tests/__init__.py` может «спрятать» тесты

Discovery проверяет найденные пакеты на наличие функции `load_tests`. Если она существует, **discovery не будет рекурсивно заходить внутрь пакета** — ответственность за сбор тестов переходит к `load_tests`. Это удобно для сложных наборов, но опасно для новичков: одна лишняя заготовка в `__init__.py` — и внезапно “0 tests”. ([Python documentation][1])

Если Вам не нужно кастомизировать загрузку — не добавляйте `load_tests` вообще.

## Типичные поломки и быстрый дебаг

**«Ran 0 tests»**
Чаще всего: файл не совпал с паттерном (`test*.py`) или имя файла не импортируемо как модуль (например, дефис), или discovery не заходит в поддиректорию без `__init__.py`. ([Python documentation][1])

**`ImportError` на импорте приложения**
Почти всегда: неверный “top level” относительно start dir, либо проект в `src/`, а `src/` не попал в `sys.path`. Напоминание: `discover()` требует, чтобы тестовые модули импортировались из top-level, и отдельно предлагает `-t/--top-level-directory`. ([Python documentation][1])

**Относительные импорты внутри тестов ломаются при запуске файла напрямую**
Относительные импорты завязаны на «имя пакета текущего модуля», а главный модуль приложения не имеет пакета — поэтому «запуск файла как скрипта» и относительные импорты плохо сочетаются. Решение: запускайте тесты через `python -m unittest ...`/discovery, а не через `python tests/unit/test_calc.py`. ([Python documentation][3])
