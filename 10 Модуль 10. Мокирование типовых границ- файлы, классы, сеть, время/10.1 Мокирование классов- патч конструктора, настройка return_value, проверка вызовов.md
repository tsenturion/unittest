# Конструктор под контролем: как мокировать классы в `unittest.mock` и не обмануть себя

Иногда unit-тест ломается не потому, что бизнес-логика сложная, а потому что внутри неё кто-то делает `SomeClient(...)`. Этот класс может ходить в сеть, открывать соединение, читать конфиг, тянуть время и превращать небольшой тест в интеграционный. Для таких случаев `unittest.mock` и существует: он позволяет временно заменить часть системы mock-объектом и потом проверить, как этот объект использовался. ([Python documentation][1])

## Введение

В разговорной речи часто говорят «замокать конструктор». Технически это не совсем точное выражение. `patch()` обычно подменяет не `__init__` как отдельную функцию, а имя класса в конкретном namespace. Когда код под тестом делает `BillingClient(...)`, он на самом деле вызывает уже не реальный класс, а созданный `patch()` mock. Экземпляр, который получает код под тестом, находится в `return_value` этого мока. Именно вокруг этой модели и строятся настройка поведения, проверки вызовов и диагностика ошибок. `patch()` работает как декоратор, context manager или class decorator и автоматически снимает подмену после выхода из области действия, даже если во время теста возникло исключение. ([Python documentation][1])

## Почему классы так часто мешают unit-тесту

Сам по себе вызов `SomeClass(...)` — не проблема. Проблема начинается там, где создание объекта уже является границей системы. Так бывает с HTTP-клиентами, адаптерами к БД, SDK внешних сервисов, классами-обёртками над файловой системой или временем. Если Ваш код внутри бизнес-функции сам создаёт такой объект, то unit-тесту приходится либо запускать реальную инфраструктуру, либо изолировать место создания. `unittest.mock` как раз предназначен для такой изоляции: Вы заменяете зависимость mock-объектом и проверяете не «настоящие побочные эффекты», а контракт взаимодействия. ([Python documentation][1])

Представьте простой сервис, который создаёт клиента биллинга и запрашивает у него счёт:

```python
# app/service.py
from app.clients import BillingClient


def issue_invoice(user_id, amount):
    client = BillingClient(base_url="https://billing.local", timeout=3)
    response = client.create_invoice(user_id=user_id, amount=amount)
    return response["invoice_id"]
```

В этом коде есть две отдельные точки интереса. Первая — сам факт создания `BillingClient` с правильными аргументами. Вторая — вызов `create_invoice()` у уже созданного объекта. Смешивать их в голове опасно. Когда Вы это делаете, тест начинает настраивать не тот объект и проверять не тот уровень вызова.

## Что именно делает `patch()` при подмене класса

Официальная документация формулирует это очень прямо: когда Вы патчите класс, класс заменяется mock-объектом, а «экземпляр» получается как результат вызова этого mock, то есть через `return_value`. Если `new` не передан, для обычной цели `patch()` создаёт `MagicMock`. Если `patch()` используется как декоратор, созданный mock передаётся в тест как дополнительный аргумент; если как context manager, он возвращается из `with`. ([Python documentation][1])

```mermaid
flowchart LR
    A[Код под тестом issue invoice] --> B[Имя BillingClient в модуле app service]
    B -->|patch заменяет имя| C[MockBillingClient]
    C -->|вызов конструктора BillingClient| D[MockBillingClient return value]
    D -->|вызов create invoice| E[instance create invoice]
    E -->|возврат результата| F[instance create invoice return value]
```

Эта схема — центральная. Если Вы видите в прод-коде `BillingClient(...)`, то проверять нужно вызов `MockBillingClient(...)`. Если видите дальше `client.create_invoice(...)`, то работать надо уже с `MockBillingClient.return_value.create_invoice`. Вся механика мокирования классов сводится к этому разделению ролей. ([Python documentation][2])

> Ключевая мысль: при «мокировании конструктора» Вы обычно не патчите `__init__` отдельно. Вы подменяете имя класса, а вызов `ClassName(...)` превращается в вызов mock-объекта. Экземпляр для настройки и проверок живёт в `return_value`. ([Python documentation][2])

## Базовый тест: патч конструктора, настройка `return_value`, проверка вызовов

Документация по `unittest.mock` показывает именно такой сценарий: patched class заменяется mock-объектом, а поведение экземпляра задаётся через `mock.return_value`. Это базовый паттерн, который Вам нужно сделать автоматическим. ([Python documentation][3])

```python
import unittest
from unittest.mock import patch

from app.service import issue_invoice


class TestIssueInvoice(unittest.TestCase):
    @patch("app.service.BillingClient")
    def test_returns_invoice_id(self, MockBillingClient):
        instance = MockBillingClient.return_value
        instance.create_invoice.return_value = {"invoice_id": "inv-001"}

        result = issue_invoice(user_id=42, amount=1500)

        self.assertEqual(result, "inv-001")
        MockBillingClient.assert_called_once_with(
            base_url="https://billing.local",
            timeout=3,
        )
        instance.create_invoice.assert_called_once_with(
            user_id=42,
            amount=1500,
        )
```

Здесь происходит три разных действия. Сначала декоратор `@patch("app.service.BillingClient")` подменяет имя `BillingClient` именно в модуле `app.service`. Затем переменная `instance` получает ссылку на mock-экземпляр через `MockBillingClient.return_value`. После этого Вы задаёте, что должен вернуть вызов `instance.create_invoice(...)`, и отдельно проверяете два слоя взаимодействия: вызов «конструктора» на уровне класса и вызов метода на уровне экземпляра. Эта логика полностью соответствует тому, как `patch()` описан в документации и в примере с мокированием классов. ([Python documentation][2])

Очень полезно запомнить простое соответствие:

| Что делает код под тестом     | Где настраивать или проверять в тесте                  |
| ----------------------------- | ------------------------------------------------------ |
| `BillingClient(...)`          | `MockBillingClient.assert_called_once_with(...)`       |
| `client = BillingClient(...)` | `instance = MockBillingClient.return_value`            |
| `client.create_invoice(...)`  | `instance.create_invoice.assert_called_once_with(...)` |
| Возврат `create_invoice(...)` | `instance.create_invoice.return_value = ...`           |

Это не «стиль написания теста», а прямое следствие устройства `patch()` и `Mock`. У mock-объектов есть специальные методы проверки вызовов, такие как `assert_called_with()` и `assert_called_once_with()`, а история вызовов хранится отдельно для самого mock и для его дочерних атрибутов. ([Python documentation][1])

Если в этот момент у Вас возникает желание написать что-то вроде `MockBillingClient.create_invoice.return_value = ...`, остановитесь. Такой код настраивает метод на mock-классе, а прод-код вызывает метод у экземпляра, то есть у `MockBillingClient.return_value`. Это две разные точки. Именно из-за их смешения появляются тесты, которые «почти правильные», но проверяют не ту трассу вызовов.

## Где проверять историю вызовов, когда одного `assert_called_once_with` мало

Для одного вызова хватает простого `assert_called_once_with()`. Но как только класс создаётся несколько раз, удобнее смотреть не только на один последний вызов, а на историю. В `unittest.mock` для этого есть `call_args_list`, а для более выразительных сравнений — helper `call` и метод `assert_has_calls()`. `call_args_list` хранит все вызовы по порядку, а `assert_has_calls()` умеет проверять ожидаемую последовательность и, при необходимости, игнорировать порядок через `any_order=True`. ([Python documentation][1])

Это особенно важно для классов, потому что конструктор и методы экземпляров часто повторяются в циклах. Тогда уже недостаточно ответа на вопрос «вызвали ли мы класс хоть раз». Вам нужен ответ на вопрос «сколько раз», «с какими аргументами», «в каком порядке» и «какой экземпляр обслужил какой сценарий».

## Самая частая ошибка: Вы патчите не там, где класс ищется

Это та ошибка, из-за которой новички теряют много времени. `patch()` подменяет имя, на которое ссылается код под тестом. Значит, патчить надо там, где объект lookup’ится, а не там, где он однажды был определён. Документация подчёркивает это отдельно: “patch where it is looked up”. Если модуль `service.py` делает `from app.clients import BillingClient`, то внутри `service.py` имя `BillingClient` уже указывает на локальную ссылку, и патч `app.clients.BillingClient` на работу `issue_invoice()` не повлияет. Патчить надо `app.service.BillingClient`. ([Python documentation][1])

Неправильный и правильный варианты выглядят так:

```python
# app/service.py
from app.clients import BillingClient


def issue_invoice(user_id, amount):
    client = BillingClient(base_url="https://billing.local", timeout=3)
    return client.create_invoice(user_id=user_id, amount=amount)
```

```python
# неверно: патчите место определения, а не lookup
@patch("app.clients.BillingClient")
def test_issue_invoice(self, MockBillingClient): ...
```

```python
# верно: патчите имя, которое использует app.service
@patch("app.service.BillingClient")
def test_issue_invoice(self, MockBillingClient): ...
```

Если же в `service.py` было бы `import app.clients` и далее использовалось `app.clients.BillingClient(...)`, тогда патчить нужно было бы `app.clients.BillingClient`. Важен не источник класса, а реальное место разрешения имени в момент выполнения кода. ([Python documentation][1])

## Когда конструктор вызывается несколько раз: `return_value` уже недостаточно

Есть тонкий момент. Если Вы просто настраиваете `MockBillingClient.return_value`, то каждый вызов `BillingClient(...)` вернёт один и тот же mock-экземпляр. Иногда это нормально. Иногда это скрывает дефект. Например, если логика должна создавать новый клиент на каждой итерации, один общий `return_value` не даст увидеть ошибку повторного использования состояния.

В `Mock` для этого предусмотрен `side_effect`. По документации он может быть функцией, исключением или итерируемым объектом. Если передать iterable, каждое обращение к mock будет брать следующее значение из последовательности. В случае patched class это позволяет раздать разные mock-экземпляры разным вызовам конструктора. Параллельно Вы можете проверить список вызовов самого класса через `call_args_list`. ([Python documentation][1])

```python
import unittest
from unittest.mock import Mock, call, patch

from app.service import batch_issue


class TestBatchIssue(unittest.TestCase):
    @patch("app.service.BillingClient")
    def test_creates_separate_clients(self, MockBillingClient):
        first = Mock(name="first_client")
        second = Mock(name="second_client")

        first.create_invoice.return_value = {"invoice_id": "inv-1"}
        second.create_invoice.return_value = {"invoice_id": "inv-2"}

        MockBillingClient.side_effect = [first, second]

        result = batch_issue([10, 20], amount=500)

        self.assertEqual(result, ["inv-1", "inv-2"])
        self.assertEqual(
            MockBillingClient.call_args_list,
            [
                call(base_url="https://billing.local", timeout=3),
                call(base_url="https://billing.local", timeout=3),
            ],
        )
        first.create_invoice.assert_called_once_with(user_id=10, amount=500)
        second.create_invoice.assert_called_once_with(user_id=20, amount=500)
```

Такой тест даёт намного более точный сигнал. Он не только подтверждает факт двух созданий клиента, но и связывает каждый вызов конструктора с конкретным экземпляром. Для сценариев с ретраями, чанками, пакетной обработкой и несколькими независимыми объектами это уже не роскошь, а нормальный уровень точности.

## Почему тест может врать: `Mock` слишком охотно создаёт атрибуты

У `Mock` и `MagicMock` есть удобное, но опасное свойство: они создают дочерние атрибуты по мере обращения к ним и запоминают, как эти атрибуты использовались. Это делает library гибкой, но одновременно позволяет написать тест, который подтвердит несуществующий API. Именно поэтому документация отдельно рекомендует использовать speccing и autospeccing, когда Вы хотите защититься от ложноположительных тестов. ([Python documentation][1])

Посмотрите на искусственно испорченный пример:

```python
# app/service.py
from app.clients import BillingClient


def issue_invoice(user_id, amount):
    client = BillingClient(base_url="https://billing.local", timeout=3)
    response = client.create_invoce(user_id=user_id, amount=amount)  # опечатка
    return response["invoice_id"]
```

```python
# плохой тест: он может пройти, хотя код сломан
@patch("app.service.BillingClient")
def test_issue_invoice(self, MockBillingClient):
    instance = MockBillingClient.return_value
    instance.create_invoce.return_value = {"invoice_id": "inv-001"}

    result = issue_invoice(user_id=42, amount=1500)

    self.assertEqual(result, "inv-001")
    instance.create_invoce.assert_called_once_with(user_id=42, amount=1500)
```

Этот тест выглядит правдоподобно. Но он проверяет не реальный контракт класса `BillingClient`, а контракт, который тест сам себе придумал. Если у настоящего класса нет метода `create_invoce`, обычный `MagicMock` всё равно позволит к нему обратиться, потому что создаст новый child mock на лету. В результате тест зелёный, а код в проде сломан. ([Python documentation][1])

## `autospec=True`: страховка от несуществующих методов и кривых сигнатур

`autospec` решает эту проблему строже. Если Вы указываете `autospec=True` в `patch()`, mock строится по спецификации заменяемого объекта. Для функций и методов это означает проверку сигнатуры вызова. Для классов — ещё и то, что `return_value`, то есть mock-экземпляр, получает ту же спецификацию, что и реальный класс. Из-за этого опечатка в названии метода превращается в `AttributeError`, а неправильные аргументы конструктора — в `TypeError`. ([Python documentation][1])

Практически это означает очень простое правило. Если Вы патчите класс, который является заметной внешней зависимостью, почти всегда стоит сначала попробовать `autospec=True`:

```python
@patch("app.service.BillingClient", autospec=True)
def test_returns_invoice_id(self, MockBillingClient):
    instance = MockBillingClient.return_value
    instance.create_invoice.return_value = {"invoice_id": "inv-001"}

    result = issue_invoice(user_id=42, amount=1500)

    self.assertEqual(result, "inv-001")
    MockBillingClient.assert_called_once_with(
        base_url="https://billing.local",
        timeout=3,
    )
    instance.create_invoice.assert_called_once_with(
        user_id=42,
        amount=1500,
    )
```

Если позже кто-то переименует `create_invoice` в `create_bill`, или изменит сигнатуру `BillingClient(...)`, такой тест перестанет «подыгрывать» сломанному коду. Он начнёт падать в точке расхождения с реальным API. Именно этого Вы и хотите от unit-теста. ([Python documentation][1])

## Где `autospec` может удивить

У `autospec` есть ограничение, которое нужно знать заранее. Он видит API класса через introspection, но не может угадать instance-атрибуты, которые создаются только внутри `__init__()`. Если реальный класс делает `self.token = ...` в конструкторе, а на уровне класса `token` не объявлен, то autospecced instance не будет «знать» про такой атрибут. Документация показывает этот случай отдельно: доступ к подобному атрибуту может привести к `AttributeError`, пока Вы не зададите его вручную. ([Python documentation][1])

Например:

```python
class BillingClient:
    def __init__(self, base_url, timeout):
        self.token = None
```

```python
@patch("app.service.BillingClient", autospec=True)
def test_uses_token(self, MockBillingClient):
    instance = MockBillingClient.return_value
    instance.token = "test-token"  # задаём явно то, что в реальном коде
    # появляется только после __init__()
    ...
```

Это не дефект `autospec`, а следствие того, как устроен Python-объект. Спецификация строится по доступным атрибутам класса, а не по тому, что когда-нибудь создастся внутри `__init__()`. Практическое правило здесь простое: если Ваш код читает такие instance-атрибуты, задавайте их явно на `return_value` или проектируйте класс так, чтобы важные поля были видны на уровне класса как значения по умолчанию. Документация даже указывает, что второй путь нередко делает код проще и быстрее. ([Python documentation][1])

## Рабочий шаблон, который стоит запомнить

Когда Вы видите в прод-коде создание зависимости через класс, держите в голове пять шагов.

1. Найдите не место определения класса, а модуль, где код под тестом реально смотрит на это имя. Именно его и нужно патчить. ([Python documentation][1])
2. Считайте patched class отдельным объектом. Его вызов — это проверка конструктора. Экземпляр — это `mock_cls.return_value`. ([Python documentation][2])
3. Настраивайте поведение методов на `mock_cls.return_value.method.return_value`, а не на самом mock-классе. Это соответствует модели из официального примера `Mocking Classes`. ([Python documentation][3])
4. Если класс создаётся несколько раз и экземпляры должны отличаться, используйте `side_effect`, а не один общий `return_value`. Историю вызовов удобно смотреть через `call_args_list` или `assert_has_calls`. ([Python documentation][1])
5. Если Вам важна защита от дрейфа API, добавляйте `autospec=True` и явно настраивайте те instance-атрибуты, которые появляются только в `__init__()`. ([Python documentation][1])

Если этот шаблон станет привычкой, мокирование классов перестанет быть магией. Вы начнёте видеть не «какую-то подмену зависимостей», а очень конкретную трассу: lookup имени, вызов patched class, получение `return_value`, вызов метода экземпляра, проверка результата.

## Заключение

Мокирование классов в `unittest.mock` кажется сложным ровно до того момента, пока Вы не разделите две сущности: mock класса и mock экземпляра. С этого момента всё становится предметным. Вызов `ClassName(...)` проверяется на уровне patched class. Поведение объекта, который вернулся из этого вызова, настраивается через `return_value`. Ошибка с namespace исчезает, как только Вы начинаете патчить место lookup, а не место определения. Ошибка с ложноположительными тестами исчезает, как только Вы подключаете `autospec` там, где контракт действительно важен. ([Python documentation][1])

Хороший тест на класс-зависимость отвечает не на один вопрос, а сразу на три. Создали ли объект правильно? Вызвали ли у него нужный метод с нужными данными? Верно ли обработали результат этого метода? Если Вы покрыли все три уровня и не забыли про `autospec`, такой тест уже не просто «изолирует внешний клиент», а действительно страхует поведение Вашего кода. ([Python documentation][3])
