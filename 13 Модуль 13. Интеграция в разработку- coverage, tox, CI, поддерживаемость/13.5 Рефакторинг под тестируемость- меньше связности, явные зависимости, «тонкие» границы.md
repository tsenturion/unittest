# Тестируемость не появляется в тестах: как рефакторить Python-код так, чтобы `unittest` проверял поведение, а не боролся с архитектурой

Когда код трудно тестировать, проблема обычно не в `unittest`. Проблема в том, что внутри одной функции смешаны вычисление, конфигурация и побочные эффекты: чтение переменных окружения, создание клиентов, работа со временем, файловой системой и сетью. В статьях про Dependency Injection и рефакторинг модульных зависимостей Мартин Фаулер отдельно подчёркивает принцип разделения конфигурации и использования зависимостей, а также важность таких границ между модулями, при которых небольшое изменение не требует держать в голове весь проект. С другой стороны, документация Python по `unittest.mock` показывает, что моки и `patch()` предназначены для замены частей системы под тестом, а не для маскировки запутанного дизайна. ([martinfowler.com][1])

## Введение

У `unittest` есть здоровая базовая модель: тест — это отдельный `TestCase`, а каждая тестовая функция получает собственную fixture. Документация прямо говорит, что для каждого тестового метода создаётся новый экземпляр `TestCase`, а `setUp()`, `tearDown()` и конструктор вызываются по одному разу на тест. Это хорошая основа для независимых проверок. Но сама по себе она не спасает от кода, который тащит глобальные зависимости, скрытые singleton-объекты и внутреннее создание клиентов. Если production-код tightly coupled, тесты почти неизбежно начинают либо чрезмерно мокировать внутренности, либо разваливаться от каждого безопасного рефакторинга. ([Python documentation][2])

Фаулер в статье про DI формулирует центральную мысль очень точно: важнее не выбор между конкретными механизмами wiring, а принцип отделения конфигурации от использования. Для тестируемости это означает простую вещь: объект не должен сам искать полмира, когда ему нужно выполнить одно действие. Чем больше зависимостей видно в сигнатуре функции или конструктора, тем меньше “магии” приходится воспроизводить в тесте. Это не абстрактная красота дизайна, а прямое уменьшение стоимости тестов. ([martinfowler.com][1])

> Хорошо тестируется не тот код, у которого много моков, а тот код, у которого мало скрытых зависимостей.

## Почему проблема обычно начинается в дизайне, а не в тестовом коде

Посмотрите на такой сервис:

```python
# app/import_user.py
from datetime import datetime, timezone
from pathlib import Path
import os

from app.http import ApiClient


def import_user(user_id: int) -> dict:
    token = os.getenv("API_TOKEN")
    if not token:
        raise RuntimeError("API_TOKEN is required")

    client = ApiClient(base_url="https://api.example.com", token=token)
    raw = client.fetch_user(user_id)

    imported_at = datetime.now(timezone.utc).isoformat()

    user = {
        "id": raw["id"],
        "name": raw["name"].strip(),
        "email": raw["email"].lower(),
        "imported_at": imported_at,
    }

    Path("audit.log").write_text(f"imported:{user_id}\n", encoding="utf-8")
    return user
```

Функция делает сразу всё: читает конфигурацию из окружения, создаёт HTTP-клиент, вызывает сеть, берёт текущее время, пишет файл и только потом формирует результат. Такой код не “плох” сам по себе в маленьком скрипте. Но как только Вы хотите стабильно тестировать его поведение, все внешние точки начинают давить одновременно: окружение, время, файловая система, сеть и конкретная форма импорта `ApiClient`. В терминах Фаулера проблема здесь именно в смешении модульных обязанностей и tangled wiring между частями программы. ([martinfowler.com][3])

Теперь посмотрите на типичный тест для такой функции:

```python
# tests/test_import_user_bad.py
import unittest
from unittest.mock import patch

from app.import_user import import_user


class TestImportUserBad(unittest.TestCase):
    @patch("app.import_user.Path.write_text")
    @patch("app.import_user.datetime")
    @patch("app.import_user.ApiClient")
    @patch("app.import_user.os.getenv")
    def test_import_user(
        self, mock_getenv, MockApiClient, mock_datetime, mock_write_text
    ):
        mock_getenv.return_value = "secret"
        MockApiClient.return_value.fetch_user.return_value = {
            "id": 7,
            "name": " Alice ",
            "email": "ALICE@EXAMPLE.COM",
        }
        mock_datetime.now.return_value.isoformat.return_value = (
            "2026-03-26T10:00:00+00:00"
        )

        result = import_user(7)

        self.assertEqual(
            result,
            {
                "id": 7,
                "name": "Alice",
                "email": "alice@example.com",
                "imported_at": "2026-03-26T10:00:00+00:00",
            },
        )
```

Такой тест формально рабочий. Но цена у него высокая. Он знает о конкретных импортированных именах, о том, что функция сама создаёт `ApiClient`, о том, что время берётся именно через `datetime.now(...).isoformat()`, и о том, что аудит пишется напрямую через `Path.write_text`. Любой безопасный рефакторинг — например, перенос логирования в отдельный объект или отказ от `Path` в пользу другого адаптера — заставит переписывать тест, даже если публичное поведение не изменилось. Это именно тот тип связи между тестом и реализацией, который Фаулер описывает как coupling tests to implementations в более mockist-стиле тестирования. А документация `patch()` дополнительно напоминает, что патч вообще работает только потому, что меняет объект, на который указывает имя в конкретном namespace; тест завязан на эту форму lookup буквально технически. ([martinfowler.com][4])

## Что в этой функции реально мешает тестируемости

Проблема не в том, что у функции “много строк”. Проблема в четырёх более точных вещах.

Во-первых, зависимости скрыты. По сигнатуре `import_user(user_id: int)` невозможно понять, что функции нужны токен, клиент, часы и канал аудита. Во-вторых, конфигурация смешана с использованием: wiring выполняется внутри самой бизнес-операции, а не в отдельной фазе сборки объекта. В-третьих, границы вокруг внешнего мира толстые и грязные: сеть, время и файл встроены прямо в логику преобразования данных. И, в-четвёртых, тест вынужден наблюдать частные детали реализации, а не только итоговый контракт. Все эти проблемы очень точно укладываются в два тезиса из источников: разделяйте конфигурацию и использование зависимостей, и проектируйте модульные границы так, чтобы локальное изменение не тянуло за собой весь код. ([martinfowler.com][1])

## Рефакторинг под тестируемость: суть без лозунгов

Если убрать модные слова, рефакторинг под тестируемость — это три движения.

Первое движение: отделить чистое вычисление от orchestration. Всё, что можно вычислить без сети, времени, файловой системы и окружения, должно жить в обычной функции или методе, который принимает данные и возвращает данные. Второе движение: сделать зависимости явными. Не читать их из воздуха, а передавать в конструктор, в функцию или в небольшой объект-обёртку. Третье движение: сделать границы тонкими. Внешний мир — это адаптеры, а не место, где живут бизнес-правила. Такой подход естественно вытекает из Fowler-style separation between configuration and use и из идеи разумных модульных границ. ([martinfowler.com][1])

Ниже — та же задача после рефакторинга.

```python
# app/import_user.py
from dataclasses import dataclass


class UserGateway:
    def fetch_user(self, user_id: int) -> dict:
        raise NotImplementedError


class Clock:
    def now_iso(self) -> str:
        raise NotImplementedError


class AuditLog:
    def record_import(self, user_id: int) -> None:
        raise NotImplementedError


def normalize_user(raw: dict, imported_at: str) -> dict:
    return {
        "id": raw["id"],
        "name": raw["name"].strip(),
        "email": raw["email"].lower(),
        "imported_at": imported_at,
    }


@dataclass
class ImportUserService:
    gateway: UserGateway
    clock: Clock
    audit_log: AuditLog

    def import_user(self, user_id: int) -> dict:
        raw = self.gateway.fetch_user(user_id)
        user = normalize_user(raw, self.clock.now_iso())
        self.audit_log.record_import(user_id)
        return user
```

Здесь произошло главное: бизнес-операция больше не занимается wiring. Она не знает, откуда берётся токен, как создаётся HTTP-клиент, в какой файл пишется аудит и кто именно поставляет текущее время. Она знает только три роли: gateway, clock и audit_log. Это и есть явные зависимости. А сама функция `normalize_user()` стала pure logic: ей не нужен mock, ей нужен вход и ожидаемый выход. Такой сдвиг полностью соответствует Fowler-принципу separating configuration from use. ([martinfowler.com][1])

## Где теперь живёт реальный wiring

После такого рефакторинга следующий естественный вопрос звучит так: “Хорошо, а кто теперь собирает реальные зависимости?” Ответ: внешний слой приложения, где wiring и должен жить. Часто это `main()`, entrypoint CLI, web-handler factory, контейнер приложения или просто кусок bootstrap-кода.

```python
# app/bootstrap.py
from datetime import datetime, timezone
from pathlib import Path
import os

from app.http import ApiClient
from app.import_user import ImportUserService, UserGateway, Clock, AuditLog


class HttpUserGateway(UserGateway):
    def __init__(self, token: str):
        self._client = ApiClient(base_url="https://api.example.com", token=token)

    def fetch_user(self, user_id: int) -> dict:
        return self._client.fetch_user(user_id)


class SystemClock(Clock):
    def now_iso(self) -> str:
        return datetime.now(timezone.utc).isoformat()


class FileAuditLog(AuditLog):
    def __init__(self, path: str):
        self._path = Path(path)

    def record_import(self, user_id: int) -> None:
        self._path.write_text(f"imported:{user_id}\n", encoding="utf-8")


def build_import_user_service() -> ImportUserService:
    token = os.getenv("API_TOKEN")
    if not token:
        raise RuntimeError("API_TOKEN is required")

    return ImportUserService(
        gateway=HttpUserGateway(token),
        clock=SystemClock(),
        audit_log=FileAuditLog("audit.log"),
    )
```

Теперь конфигурация собрана в одном месте. Business-код не читает окружение и не создаёт инфраструктуру сам. Это не только проще тестировать. Это проще менять: новый способ логирования, другой клиент, подмена времени, иной путь к файлу — всё это меняется в bootstrap-слое, а не размазывается по доменной операции. Именно о такой модульной структуре Фаулер пишет в эссе про refactoring module dependencies: хороший разрез кода нужен затем, чтобы маленькое изменение не требовало понимать и переписывать весь модульный граф. ([martinfowler.com][3])

> Чем ближе wiring к краю приложения, тем легче тестировать центр системы как обычный код, а не как набор побочных эффектов.

## Явные зависимости: какие формы реально работают в Python

В статье про Dependency Injection Фаулер перечисляет несколько форм инъекции, включая constructor injection и setter injection. В Python в повседневной практике чаще всего хватает двух форм: передачи зависимостей в конструктор и передачи зависимостей параметрами функции. Иногда нужен и третий вариант — небольшой boundary-object, который инкапсулирует группу связанных внешних действий. Но суть остаётся той же: зависимость видна явно в API, а не спрятана внутри метода. ([martinfowler.com][1])

Выглядит это так:

```python
# 1. Конструктор
service = ImportUserService(gateway, clock, audit_log)

# 2. Параметр функции
def import_user(raw, *, now_iso, record_import):
    ...

# 3. Небольшой boundary-object
class Notifications:
    def send_email(self, ...): ...
    def publish_event(self, ...): ...
```

Конструктор хорош, когда зависимость нужна на протяжении всей жизни объекта. Параметр функции хорош, когда операция маленькая и объект ради неё избыточен. Boundary-object полезен, когда несколько внешних действий логически относятся к одному слою и их удобно держать рядом. Это уже не отдельное правило из стандарта Python, а практический вывод из общего DI-принципа: главное, чтобы зависимости были явными и конфигурация не пряталась в теле use-case. ([martinfowler.com][1])

## Тонкие границы: что это значит в коде

Фраза “тонкая граница” легко звучит слишком абстрактно. В практическом коде это значит очень простую вещь: boundary-класс или boundary-функция должны делать перевод между внешним миром и Вашей моделью, а не принимать внутри себя бизнес-решения. `HttpUserGateway` должен сходить в сеть и вернуть данные. `SystemClock` должен дать текущее время. `FileAuditLog` должен записать событие. Но правило “какой именно пользователь считается валидным”, “как нормализуется email” и “что делать после повторного запроса” лучше держать за пределами границы. Иначе внешний адаптер снова превращается в труднотестируемый комбайн. Это напрямую согласуется с Fowler-идеей small module boundaries и with separation of configuration/use. ([martinfowler.com][3])

Полезно видеть это через короткую таблицу:

| Внешний источник | Плохая форма                                     | Лучшая форма                            |
| ---------------- | ------------------------------------------------ | --------------------------------------- |
| HTTP/API         | use-case сам создаёт клиента и парсит всё внутри | use-case зависит от `gateway.fetch_*()` |
| Время            | код сам зовёт `datetime.now()` в разных местах   | зависимость `clock.now_*()`             |
| Файлы            | доменная функция пишет в `Path(...)` напрямую    | отдельный `audit_log` / `storage`       |
| `os.environ`     | бизнес-метод читает env сам                      | bootstrap читает env и собирает сервис  |

Эта таблица — уже инженерное обобщение. Но оно хорошо поддерживается и DI-подходом Фаулера, и самим устройством `unittest.mock`: тонкие границы как раз и дают те seams, где patching и boundary-level mocks наиболее уместны. ([martinfowler.com][1])

## Как после такого рефакторинга меняются тесты

Вот тот же сценарий, но уже в нормальном `unittest`-стиле с простыми fake-объектами вместо цепочки patch-декораторов.

```python
# tests/test_import_user.py
import unittest

from app.import_user import ImportUserService


class FakeGateway:
    def __init__(self, raw):
        self.raw = raw
        self.calls = []

    def fetch_user(self, user_id: int) -> dict:
        self.calls.append(user_id)
        return self.raw


class FixedClock:
    def __init__(self, value: str):
        self.value = value

    def now_iso(self) -> str:
        return self.value


class SpyAuditLog:
    def __init__(self):
        self.imported = []

    def record_import(self, user_id: int) -> None:
        self.imported.append(user_id)


class TestImportUserService(unittest.TestCase):
    def setUp(self):
        self.gateway = FakeGateway(
            {"id": 7, "name": " Alice ", "email": "ALICE@EXAMPLE.COM"}
        )
        self.clock = FixedClock("2026-03-26T10:00:00+00:00")
        self.audit_log = SpyAuditLog()
        self.service = ImportUserService(
            gateway=self.gateway,
            clock=self.clock,
            audit_log=self.audit_log,
        )

    def test_import_user_returns_normalized_payload(self):
        result = self.service.import_user(7)

        self.assertEqual(
            result,
            {
                "id": 7,
                "name": "Alice",
                "email": "alice@example.com",
                "imported_at": "2026-03-26T10:00:00+00:00",
            },
        )
        self.assertEqual(self.gateway.calls, [7])
        self.assertEqual(self.audit_log.imported, [7])
```

В этом тесте заметно сразу несколько улучшений. Во-первых, нет patching временных имён в модуле. Во-вторых, нет скрытой магии времени, окружения и файлов. В-третьих, fake-объекты очень дешёвые: они не имитируют внутренности сторонней библиотеки, а реализуют только тот маленький контракт, который действительно нужен use-case. И, наконец, `setUp()` здесь снова делает то, что и должен делать fixture hook в `unittest`: собирает локальную тестовую среду для одного `TestCase` instance, а не чинит архитектурные дыры production-кода. Документация `unittest` как раз и описывает fixture как подготовку окружения и cleanup для тестов, причём отдельный экземпляр `TestCase` создаётся на каждый тестовый метод. ([Python documentation][2])

## Почему fake здесь лучше, чем mock

В статье `Mocks Aren't Stubs` Фаулер разводит state verification и behavior verification. Для “простых коллабораторов” классический стиль часто оказывается лучше: Вы даёте системе под тестом реальный или почти реальный объект и проверяете итоговое состояние или результат. Именно это и происходит в примере выше. `FakeGateway`, `FixedClock` и `SpyAuditLog` не пытаются подробно контролировать внутреннюю хореографию. Они просто дают детерминированный внешний мир, в котором use-case можно честно прогнать. Такой тест обычно устойчивее к безопасным внутренним перестройкам и лучше совпадает с тем, что Вам реально важно в поведении сервиса. ([martinfowler.com][4])

Это не значит, что `Mock` больше не нужен. Это значит только, что у него должна быть причина. Если зависимость дешёвая и детерминированная, fake часто делает тест проще и честнее. Если зависимость дорогая, нестабильная или внешняя, тогда mock уместен. Фаулер именно так и описывает различие между стилями: вопрос не в “верить ли в моки вообще”, а в том, где поведенческая верификация действительно добавляет ценность, а где она слишком крепко привязывает тест к реализации. ([martinfowler.com][4])

## Когда patch всё ещё правильный инструмент

После такого разговора легко впасть в другую крайность и решить, что `patch()` — это всегда smell. Это неверно. Документация `unittest.mock` описывает `patch()` как штатный способ временно заменить объект, на который указывает имя в конкретном namespace. Более того, она отдельно подчёркивает главное правило: патчить нужно там, где объект **ищется** системой под тестом, а не там, где он был определён. Для реальных внешних границ это очень полезно. Например, когда Вы тестируете тонкий HTTP-адаптер, а не доменный сервис, patch на клиентскую библиотеку или на конструктор API-класса вполне уместен. ([Python documentation][5])

Вот пример boundary-level теста:

```python
# app/bootstrap.py
class HttpUserGateway(UserGateway):
    def __init__(self, token: str):
        self._client = ApiClient(base_url="https://api.example.com", token=token)

    def fetch_user(self, user_id: int) -> dict:
        return self._client.fetch_user(user_id)
```

```python
# tests/test_bootstrap.py
import unittest
from unittest.mock import patch

from app.bootstrap import HttpUserGateway


class TestHttpUserGateway(unittest.TestCase):
    @patch("app.bootstrap.ApiClient", autospec=True)
    def test_fetch_user_delegates_to_api_client(self, MockApiClient):
        MockApiClient.return_value.fetch_user.return_value = {"id": 7}

        gateway = HttpUserGateway("secret")
        result = gateway.fetch_user(7)

        self.assertEqual(result, {"id": 7})
        MockApiClient.assert_called_once_with(
            base_url="https://api.example.com",
            token="secret",
        )
        MockApiClient.return_value.fetch_user.assert_called_once_with(7)
```

Здесь `patch()` уже тестирует не use-case, а конкретный boundary-object. И в таком месте это нормально: весь смысл класса в том, чтобы правильно провайрить внешний клиент и правильно делегировать вызов. Плюс здесь уместен `autospec=True`. Документация `unittest.mock` говорит, что autospec создаёт mock с той же API-поверхностью, что и заменяемый объект, а функции и методы будут проверять сигнатуру и поднимать `TypeError` при неверном вызове. Для класса это распространяется и на его `return_value`, то есть на mock-экземпляр. Это полезно на границах, потому что защищает тест от опечаток и незаметного дрейфа API. ([Python documentation][5])

Документация также отдельно предупреждает: обычный mock лениво создаёт атрибуты по мере обращения. Это удобно, но и опасно — можно написать тест против метода, которого в реальном объекте нет. Autospec и `create_autospec()` как раз нужны для того, чтобы такие ошибки ловились рано. То есть правило здесь очень простое: в доменном коде предпочитайте явные зависимости и fakes, а на тонких внешних границах, где patch действительно нужен, делайте его строже через autospec. ([Python documentation][5])

## Что такое “хорошая граница” в реальном проекте

Хорошая граница отвечает на два вопроса.

Первый: можно ли заменить эту зависимость простым fake-объектом, не переписывая тест на пол-инфраструктуры? Если да, это уже хороший знак. Второй: понятна ли обязанность адаптера из одного-двух предложений? Например: “получить пользователя из внешнего API”, “сказать текущее время”, “сохранить audit-событие”, “прочитать конфиг”. Если Вам нужно полстраницы объяснять, что делает boundary-класс, он почти наверняка уже содержит доменную логику, которую лучше вытянуть внутрь системы. Эта эвристика — следствие идеи thin boundaries, а не отдельное правило документации. Но она хорошо сочетается с Fowler-подходом к модульным зависимостям и с тем, как `unittest.mock` лучше всего работает в реальных тестах. ([martinfowler.com][3])

## Признаки, что код пора рефакторить под тестируемость

Ниже — короткая практическая карта. Это не “формальные запреты”, а частые сигналы того, что дизайн уже тянет тесты вниз.

| Симптом в тесте                                                                 | Что это обычно значит в коде                           | Какой рефакторинг помогает               |
| ------------------------------------------------------------------------------- | ------------------------------------------------------ | ---------------------------------------- |
| 4–6 `@patch(...)` на один тест                                                  | use-case сам собирает инфраструктуру                   | вынести wiring наружу                    |
| нужно patch private helper                                                      | тест знает о внутренней декомпозиции                   | проверять публичный результат            |
| приходится мокировать `datetime`, `os.getenv`, `Path`, HTTP-клиент одновременно | функция смешивает бизнес-логику и I/O                  | выделить gateway/clock/storage           |
| mock-цепочки вида `x.return_value.y.return_value...`                            | зависимость слишком толстая или слишком скрытая        | сделать boundary тоньше                  |
| один и тот же тяжёлый setup повторяется в тестах                                | нет нормального объекта-сервиса с явными зависимостями | ввести use-case object / fixture builder |

Эти признаки хорошо согласуются и с Fowler-описанием coupling tests to implementations, и с самой механикой `patch()`/`Mock` в `unittest.mock`. Чем больше тест знает о частной wiring-схеме production-кода, тем вероятнее, что архитектура уже просит рефакторинга. ([martinfowler.com][4])

## Пошаговая стратегия рефакторинга, которая обычно работает

Когда код уже большой, переписать всё “по красоте” редко реально. Намного лучше работает короткий итеративный маршрут.

Сначала найдите побочные эффекты. Время, env, сеть, файл, БД, очередь, глобальный реестр — всё это кандидаты на границы. Потом отделите чистое вычисление от orchestration: что можно проверить обычным `assertEqual()` без patch и без инфраструктуры? Затем введите явные зависимости — сначала хотя бы параметрами функции, потом, если нужно, конструктором сервиса. После этого соберите реальную конфигурацию в одном внешнем месте: bootstrap, factory, `main()`, контейнер, DI-модуль — название не так важно. Фаулер именно на этом настаивает как на ключевом принципе: конфигурация и использование не должны быть спаяны в одном месте. ([martinfowler.com][1])

Очень важно не делать этот процесс религиозным. В Python не нужно вводить десятки интерфейсов только затем, чтобы “соответствовать DI-стилю”. Если зависимость удобно передать как функцию — передайте функцию. Если нужен маленький объект с одним методом — сделайте маленький объект. Ценность не в количестве абстракций, а в явности и локальности зависимостей. Это уже инженерный вывод из общего DI-принципа, а не отдельная догма из стандарта. ([martinfowler.com][1])

## Где здесь место `unittest`

После такого рефакторинга `unittest` снова начинает работать в свою силу. `setUp()` и `tearDown()` действительно собирают и разбирают fixture, а не компенсируют скрытый wiring. Отдельный экземпляр `TestCase` на каждый тест помогает держать состояние локальным. `assertEqual()` и другие типоспецифические assert’ы дают нормальные diff. А `patch()` остаётся там, где ему и место: на тонких границах, при тестировании внешних адаптеров и редких integration-like seams. Именно в такой конфигурации testing framework перестаёт быть “местом борьбы с архитектурой” и становится нормальным способом проверить поведение. ([Python documentation][2])

## заключение

Рефакторинг под тестируемость — это не про “написать больше тестов”. Это про то, чтобы код сам допускал честные, короткие и устойчивые проверки. Меньше связности означает меньше скрытого wiring внутри use-case. Явные зависимости означают, что нужные коллабораторы видны в API, а не добываются из воздуха. Тонкие границы означают, что сеть, время, файлы и окружение вынесены в адаптеры, которые легко заменить fake-объектом или, если нужно, строго патчить через `autospec`. Именно такой дизайн даёт тестам шанс проверять поведение, а не внутреннюю машинерию. ([martinfowler.com][1])

Если свести всё к одной практической формуле, она будет такой: **сначала сделайте зависимости видимыми, потом вынесите wiring на край приложения, и только после этого решайте, где нужен mock, а где достаточно fake**. Тогда `unittest` начинает работать по назначению: каждый тест получает свою локальную fixture, assertions объясняют расхождения, а patch остаётся точным инструментом на границах, а не универсальной заплаткой на архитектурные проблемы. ([Python documentation][2])
