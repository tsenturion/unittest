# `patch.dict()` без сюрпризов: как временно менять словари и `os.environ`, чтобы тесты не зависели от среды

Вы запускаете тест локально, и он зелёный. В CI тот же тест падает. Причина нередко банальна: код читает настройки из словаря или из переменных окружения, а тест незаметно зависит от того, что уже лежит в процессе. В таких случаях `patch()` для функций и методов — не лучший инструмент. Правильная точка подмены здесь не функция, а **mapping**: обычный словарь, `os.environ`, `sys.modules` или ваш собственный словареподобный контейнер. Для этой задачи в `unittest.mock` есть отдельный инструмент — `patch.dict()`. Он временно меняет содержимое словаря или словареподобного объекта и по завершении теста восстанавливает прежнее состояние. ([Python documentation][1])

Снаружи `patch.dict()` выглядит скромно. Но в модуле про `patch*` это один из самых практичных приёмов. Через него удобно тестировать feature flags, конфигурацию, поведение по умолчанию при отсутствии ключа, переменные окружения и даже импорт необязательных зависимостей. Причём библиотека поддерживает три формы использования: context manager, decorator и class decorator. Это значит, что Вы можете выбрать точную область действия подмены, не оставляя «грязь» после теста. ([Python documentation][1])

## Введение

Тема `patch.dict()` кажется узкой только до первого реального проекта. Как только приложение начинает жить дольше учебного примера, в нём появляются словари с настройками, реестр подключённых адаптеров, кеши, `sys.modules`, `os.environ`, а иногда и собственные контейнеры конфигурации. И если код под тестом читает значения из такого источника, то самый честный способ проверить ветвление — временно изменить **сам источник данных**, а не имитировать чтение косвенными патчами. Именно это и делает `patch.dict()`: он работает с mapping-объектами, а не с произвольными атрибутами. ([Python documentation][1])

В официальной документации сигнатура выглядит так: `patch.dict(in_dict, values=(), clear=False, **kwargs)`. Здесь важно почти всё. `in_dict` может быть не только обычным словарём, но и словареподобным контейнером. `values` может быть словарём или итерируемой последовательностью пар `(key, value)`. Дополнительно значения можно передавать и keyword-аргументами. Параметр `clear=False` означает «наложить новые ключи поверх существующих», а `clear=True` означает «сначала очистить mapping, потом заполнить новыми значениями». Наконец, при использовании как context manager, начиная с Python 3.8, `patch.dict()` возвращает сам patched-словарь. ([Python documentation][1])

Это небольшое API, но от него зависит архитектурная чистота теста. Если тест проверяет логику выбора режима по конфигурации, подменять надо не `if` и не `os.getenv`, а **данные**, по которым выполняется выбор. Тогда тест проверяет реальный сценарий, а не искусственно сконструированную обёртку вокруг сценария. Такой подход особенно ценен в unit-тестах, где Вы хотите изолировать бизнес-правило, но не хотите подменять полмира ради одного флага. ([Python documentation][1])

## Что `patch.dict()` делает на самом деле

Полезно мыслить о `patch.dict()` не как о «магии для словаря», а как о временной транзакции над mapping-объектом. Перед входом в тестовую область библиотека запоминает исходное содержимое, затем применяет Ваши изменения, а после выхода восстанавливает словарь к прежнему состоянию. Документация специально уточняет, что восстановление идёт к исходному состоянию словаря, причём восстановленная версия — это копия состояния «до теста». Внутри области подмены Вы можете не только читать значения, но и свободно добавлять, обновлять и удалять ключи: при выходе из `with` исходное состояние всё равно будет восстановлено. ([Python documentation][1])

Минимальный пример выглядит так:

```python
from unittest.mock import patch

settings = {"mode": "prod", "timeout": 30}

with patch.dict(settings, {"mode": "test"}):
    assert settings["mode"] == "test"
    assert settings["timeout"] == 30

assert settings == {"mode": "prod", "timeout": 30}
```

Здесь произошло не «копирование нового словаря поверх старого навсегда», а временное наложение изменений. Важная деталь: по умолчанию `clear=False`, поэтому несвязанные ключи вроде `timeout` сохраняются. Это и делает `patch.dict()` удобным, когда тесту нужно поменять только один-два флага, не разрушая весь остальной контекст. ([Python documentation][1])

```mermaid
flowchart LR
    A[Исходный mapping] --> B[Снимок состояния]
    B --> C{clear?}
    C -->|False| D[Существующие ключи остаются]
    C -->|True| E[Mapping очищается]
    D --> F[Накладываются новые значения]
    E --> F
    F --> G[Код под тестом читает и меняет mapping]
    G --> H[Выход из decorator/with]
    H --> I[Восстановление исходного состояния]
```

Эта схема помогает увидеть главный смысл `patch.dict()`: он не подменяет ссылку на объект, а временно меняет **содержимое** mapping-источника и затем откатывает его назад. Это особенно важно для `os.environ`, потому что код в разных частях программы может держать ссылку на один и тот же объект `os.environ`; временная правка содержимого в таких случаях полезнее, чем подмена обёртки вокруг чтения. ([Python documentation][1])

Начиная с Python 3.8, context manager возвращает patched-словарь, и этим можно пользоваться для более читаемого Arrange-кода:

```python
from unittest.mock import patch

flags = {}

with patch.dict(flags, {"beta": "1"}) as patched_flags:
    patched_flags["trace"] = "enabled"
    assert flags["beta"] == "1"
    assert flags["trace"] == "enabled"

assert flags == {}
```

Это не другой объект: `patched_flags` и `flags` указывают на один и тот же mapping, просто в текущем patched-состоянии. После выхода из блока он восстанавливается. Для теста это удобно: можно явно показать, что внутри области подмены Вы работаете с тем же источником конфигурации, который читает production-код. ([Python documentation][1])

## `clear=False` и `clear=True` — это два разных сценария

Многие ошибки с `patch.dict()` начинаются не с синтаксиса, а с неправильного режима. По умолчанию `clear=False`. Это режим «частичного наложения»: старые ключи остаются, а новые и изменённые просто обновляют mapping. Когда Вы тестируете один feature flag в уже существующей конфигурации, это обычно именно то, что нужно. Документация описывает этот режим как обычное добавление или изменение значений в словаре на время теста. ([Python documentation][1])

`clear=True` — это уже другой тип теста. Здесь mapping сначала очищается, и только потом в него попадают переданные значения. Такой режим нужен, когда поведение зависит не просто от значения ключа, а от **самого факта присутствия или отсутствия** ключа. Особенно часто это бывает с `os.environ`: одна ветка кода срабатывает при наличии `APP_MODE`, другая — при его отсутствии и использовании значения по умолчанию. Если оставить `clear=False`, тест может незаметно унаследовать посторонние переменные окружения и пройти не потому, что логика верна, а потому, что окружение на вашей машине «подходящее». Документация показывает `clear=True` именно как режим полной очистки перед установкой новых значений. ([Python documentation][1])

Полезное практическое правило здесь простое.

> Если тест проверяет **замену** конкретного ключа, обычно подходит `clear=False`.
> Если тест проверяет **отсутствие** ключей, дефолты и «чистое» окружение, почти всегда нужен `clear=True`.

Разницу удобно увидеть на маленьком примере:

```python
from unittest.mock import patch

SETTINGS = {
    "mode": "prod",
    "timeout": 30,
    "retries": 3,
}

with patch.dict(SETTINGS, {"mode": "test"}):
    assert SETTINGS == {
        "mode": "test",
        "timeout": 30,
        "retries": 3,
    }

with patch.dict(SETTINGS, {"mode": "test"}, clear=True):
    assert SETTINGS == {
        "mode": "test",
    }
```

В первом случае Вы моделируете «тот же мир, но с другим режимом». Во втором — «чистый мир, в котором существует только то, что я явно задал». Для unit-тестов на конфигурацию это две принципиально разные постановки задачи. ([Python documentation][1])

## Обычный словарь конфигурации: самый прямой сценарий

Начать лучше с простого. Допустим, сервис читает настройки из модульного словаря:

```python
# pricing.py
PRICING_RULES = {
    "discount_enabled": False,
    "discount_percent": 0,
    "currency": "RUB",
}


def final_price(base_price: int) -> int:
    if not PRICING_RULES["discount_enabled"]:
        return base_price
    discount = base_price * PRICING_RULES["discount_percent"] // 100
    return base_price - discount
```

Тесты здесь естественно писать через `patch.dict()`, потому что источник решения — это именно словарь `PRICING_RULES`, а не функция-обёртка:

```python
# test_pricing.py
import unittest
from unittest.mock import patch

from pricing import PRICING_RULES, final_price


class TestFinalPrice(unittest.TestCase):
    def test_returns_base_price_when_discount_disabled(self):
        with patch.dict(
            PRICING_RULES, {"discount_enabled": False, "discount_percent": 15}
        ):
            self.assertEqual(final_price(1000), 1000)

    def test_applies_discount_when_enabled(self):
        with patch.dict(
            PRICING_RULES, {"discount_enabled": True, "discount_percent": 15}
        ):
            self.assertEqual(final_price(1000), 850)
```

Такой тест хорош по двум причинам. Во-первых, он меняет только значимую часть состояния. Во-вторых, после завершения каждый тест гарантированно оставляет `PRICING_RULES` в исходном виде. Это соответствует documented-поведению `patch.dict()` как временной правки mapping-объекта с восстановлением состояния после выхода из области подмены. ([Python documentation][1])

В этом месте многие совершают лишнее движение и начинают патчить функцию `final_price`, helper-функцию чтения правил или геттер поверх словаря. Обычно это сигнал, что Вы теряете главный критерий выбора инструмента: `patch.dict()` нужен тогда, когда код принимает решение по значениям из mapping-источника. Если источник именно словарь, то и патчить логично словарь. Если же конфигурация хранится как обычный атрибут объекта, нужен уже `patch()` или `patch.object()`, а не `patch.dict()`. Документация `patch.dict()` ограничивает его работу словарями и словареподобными контейнерами. ([Python documentation][1])

## Почему `os.environ` — главный практический объект для `patch.dict()`

Самый частый боевой сценарий — не произвольный словарь, а `os.environ`. В документации `os` он описан как mapping, где ключи и значения — строки, представляющие окружение текущего процесса. Есть несколько тонкостей, которые важно знать до написания тестов. `os.environ` захватывается при первом импорте модуля `os`, обычно ещё на старте Python. Изменения, сделанные **через сам `os.environ`**, видны этому mapping; библиотека автоматически вызывает `putenv()` при модификации и `unsetenv()` при удалении элементов. На Windows имена ключей приводятся к верхнему регистру. ([Python documentation][2])

Из этого следует очень практический вывод: если код под тестом читает окружение, обычно правильнее патчить именно `os.environ`, а не `os.getenv` и не `os.putenv`. Документация прямо говорит, что `os.getenv()` использует `os.environ`, а вызов `os.putenv()` напрямую **не** меняет `os.environ`; поэтому лучше модифицировать `os.environ` непосредственно. Это делает `patch.dict('os.environ', ...)` предпочтительным инструментом для большинства env-тестов: он одновременно покрывает код, который читает через `os.environ["KEY"]`, и код, который читает через `os.getenv("KEY")`. ([Python documentation][2])

Есть и платформенный нюанс. На Windows ключи в `os.environ` нормализуются в верхний регистр. Значит, кроссплатформенный тест стоит писать с верхнерегистровыми именами переменных: `APP_MODE`, `DB_HOST`, `FEATURE_X`. Тогда поведение будет одинаково читаемо на Linux, macOS и Windows, и Вы не будете ловить скрытую неоднозначность вокруг `app_mode` против `APP_MODE`. Эта особенность прямо описана в документации `os`. ([Python documentation][2])

Если окружение меняется **вне Python** или через прямые вызовы `os.putenv()`/`os.unsetenv()`, стандартная библиотека с Python 3.14 предоставляет `os.reload_environ()`, который обновляет кеш `os.environ` и `os.environb`. Но у этого API есть важное ограничение: он не thread-safe. Для обычных unit-тестов это, как правило, не нужно, потому что `patch.dict()` меняет `os.environ` напрямую, а именно такие изменения документированы как видимые самому mapping-объекту. ([Python documentation][2])

Наконец, есть важная граница применимости. Если тестируемый код запускает дочерний процесс и **явно** передаёт параметр `env=` в `subprocess.run()` или `Popen()`, то этот mapping используется **вместо** наследования текущего окружения процесса. В таком сценарии `patch.dict(os.environ, ...)` может вообще не повлиять на дочерний процесс, если код под тестом строит отдельный словарь `env` без оглядки на текущее `os.environ`. То есть `patch.dict()` по-прежнему полезен для логики внутри текущего процесса, но он уже не единственный источник истины для subprocess-сценариев. ([Python documentation][3])

## Полный пример: тестируем код, который читает окружение

Посмотрите на модуль, который собирает имена ресурсов из env:

```python
# storage.py
import os


def upload_bucket() -> str:
    region = os.getenv("APP_REGION", "eu-central-1")
    stage = os.getenv("APP_STAGE", "prod")
    return f"{stage}-{region}-uploads"


def s3_endpoint() -> str:
    scheme = os.getenv("APP_SCHEME", "https")
    host = os.environ["APP_S3_HOST"]
    return f"{scheme}://{host}"
```

Здесь уже видно, почему выгодно патчить именно `os.environ`: одна функция использует `os.getenv()`, другая — прямой доступ через `os.environ[...]`. Один источник подмены закрывает оба случая, потому что `getenv()` опирается на тот же mapping. ([Python documentation][2])

Тесты могут выглядеть так:

```python
# test_storage.py
import unittest
from unittest.mock import patch

from storage import s3_endpoint, upload_bucket


class TestStorage(unittest.TestCase):
    @patch.dict("os.environ", {"APP_REGION": "eu-west-1", "APP_STAGE": "test"})
    def test_upload_bucket_reads_environment(self):
        self.assertEqual(upload_bucket(), "test-eu-west-1-uploads")

    def test_upload_bucket_uses_defaults_when_env_is_absent(self):
        with patch.dict("os.environ", {}, clear=True):
            self.assertEqual(upload_bucket(), "prod-eu-central-1-uploads")

    def test_s3_endpoint_can_be_configured_with_keyword_args(self):
        with patch.dict(
            "os.environ", APP_SCHEME="http", APP_S3_HOST="localhost:9000", clear=True
        ):
            self.assertEqual(s3_endpoint(), "http://localhost:9000")
```

В этом фрагменте показаны сразу три полезных приёма. Во-первых, `patch.dict()` принимает строку `'os.environ'`, то есть имя словаря, которое библиотека получает через импорт. Во-вторых, для env-значений можно использовать не только словарь, но и keyword-аргументы. В-третьих, `clear=True` здесь уместен во втором и третьем тестах, потому что они хотят полностью контролировать среду, а не надеяться, что где-то не осталось подходящих переменных от другого теста, локального shell или CI-агента. Все эти возможности описаны в официальной документации `unittest.mock`. ([Python documentation][1])

Этот пример одновременно показывает важный принцип изоляции: тест подменяет **входные данные окружения**, а не реализацию функции чтения окружения. Поэтому код проходит тот же путь, что и в продакшене. Для конфигурационных тестов это почти всегда лучше, чем патчить `os.getenv` отдельным mock-объектом. Когда функция под тестом может читать и через `getenv`, и через `os.environ`, патчинг только `os.getenv` создаёт неполную модель реального сценария. Документация `os` позволяет сделать более прямую подмену, потому что `getenv()` использует `os.environ`. ([Python documentation][2])

## Самая частая ловушка: переменная окружения прочитана при импорте модуля

Теперь к сценарию, который ломает очень много тестов. Допустим, модуль вычисляет константу при импорте:

```python
# flags.py
import os

BETA_ENABLED = os.getenv("BETA_ENABLED", "0") == "1"


def use_beta_flow() -> bool:
    return BETA_ENABLED
```

Если Вы импортировали `flags`, а потом внутри теста сделали `patch.dict("os.environ", {"BETA_ENABLED": "1"})`, переменная `BETA_ENABLED` сама по себе не пересчитается. Это не ошибка `patch.dict()`. Это обычная семантика модулей: код верхнего уровня выполняется при импорте модуля, а затем значение уже лежит в пространстве имён модуля как обычная переменная. Документация по модулям прямо говорит, что исполняемые инструкции модуля запускаются при первом импорте, а `importlib.reload()` повторно выполняет module-level code и обновляет имена в словаре модуля. ([Python documentation][4])

Правильный тест в такой ситуации выглядит так:

```python
# test_flags.py
import importlib
import unittest
from unittest.mock import patch

import flags


class TestFlags(unittest.TestCase):
    def test_beta_enabled_after_reload_inside_patch(self):
        with patch.dict("os.environ", {"BETA_ENABLED": "1"}):
            importlib.reload(flags)
            self.assertTrue(flags.use_beta_flow())
```

Такой тест работает, потому что `reload()` повторно выполняет модульный код, и выражение `os.getenv("BETA_ENABLED", "0") == "1"` вычисляется уже внутри patched-окружения. Но здесь есть ещё один слой тонкости. Документация `importlib.reload()` отдельно предупреждает: если другой модуль сделал `from flags import BETA_ENABLED`, то reload модуля `flags` **не** перебиндует это имя в чужом namespace автоматически. Это важное напоминание о том, что env-конфигурация, вычисляемая при импорте, делает тесты более хрупкими. ([Python documentation][5])

Практический вывод отсюда жёсткий, но полезный: если конфигурацию можно читать не при импорте, а в функции или в явном объекте настроек, тестировать её будет проще. `patch.dict()` отлично работает с `os.environ`, но он не «переписывает историю» и не может автоматически пересчитать значения, которые модуль уже успел вычислить и закешировать в своих глобальных переменных. Это не ограничение инструмента, а обычное следствие того, как Python выполняет модульный код. ([Python documentation][4])

## `sys.modules`: продвинутый, но очень полезный сценарий

Есть ещё один случай, где `patch.dict()` оказывается сильнее, чем ожидают. Это временная правка `sys.modules`. В примерах официальной документации `unittest.mock` показано, что импорт получает объект из словаря `sys.modules`. Поэтому, если временно положить туда mock или фальшивый модуль под нужным именем, импорт во время patched-области возьмёт именно его. Документация приводит это как способ повлиять на local import внутри функции и подчёркивает, что local imports вообще сложнее для мокирования. ([Python documentation][6])

Например:

```python
# serializer.py
def dumps(payload: dict) -> bytes:
    import fastjson  # локальный импорт

    return fastjson.dumps(payload)
```

Тест:

```python
# test_serializer.py
import unittest
from unittest.mock import Mock, patch

from serializer import dumps


class TestSerializer(unittest.TestCase):
    def test_dumps_uses_fake_optional_dependency(self):
        fake_fastjson = Mock()
        fake_fastjson.dumps.return_value = b'{"ok": true}'

        with patch.dict("sys.modules", {"fastjson": fake_fastjson}):
            result = dumps({"ok": True})

        self.assertEqual(result, b'{"ok": true}')
        fake_fastjson.dumps.assert_called_once_with({"ok": True})
```

Этот приём особенно полезен для необязательных зависимостей и локальных импортов. Пока действует `patch.dict("sys.modules", ...)`, импорт получает подставленный объект. Когда область патча заканчивается, прежнее состояние `sys.modules` восстанавливается. Документация `mock-examples` показывает и более сложные варианты — например, подмену package import через несколько записей в `sys.modules`. ([Python documentation][7])

Но здесь важно не увлечься. В той же документации сказано, что local imports обычно лучше избегать: их часто используют либо для обхода циклических зависимостей, либо для отложенной загрузки, и обе проблемы чаще решаются более чистым дизайном. `patch.dict("sys.modules", ...)` — полезный инструмент, но не оправдание для тяжёлой архитектуры, где вся подменяемость строится только вокруг импорта. ([Python documentation][6])

## `patch.dict()` работает не только с `dict`

Это один из самых недооценённых фактов про API. В документации прямо сказано, что `patch.dict()` умеет работать не только с настоящими словарями, но и со словареподобными объектами. Минимальные требования такие: объект должен поддерживать получение, установку и удаление элементов, а также либо итерацию по ключам, либо проверку вхождения. На уровне протокола это `__getitem__`, `__setitem__`, `__delitem__` и либо `__iter__`, либо `__contains__`. ([Python documentation][1])

Это открывает полезный путь для тестов, если у Вас есть собственная обёртка над конфигурацией:

```python
from unittest.mock import patch


class SettingsBox:
    def __init__(self):
        self._data = {"mode": "prod"}

    def __getitem__(self, key):
        return self._data[key]

    def __setitem__(self, key, value):
        self._data[key] = value

    def __delitem__(self, key):
        del self._data[key]

    def __iter__(self):
        return iter(self._data)


box = SettingsBox()

with patch.dict(box, mode="test", debug="1"):
    assert box["mode"] == "test"
    assert box["debug"] == "1"

assert box["mode"] == "prod"
```

Это важный момент для реальных систем. Не нужно переписывать собственный контейнер настроек в «обычный dict только ради тестов». Если объект соблюдает mapping-протокол, `patch.dict()` уже подходит. Для кода базы это удобно: production-контейнер остаётся тем же, а тест меняет только содержимое на ограниченное время. Документация приводит похожий пример с пользовательским контейнером. ([Python documentation][1])

## Когда class decorator действительно уместен

`patch.dict()` можно использовать как class decorator. В этом режиме библиотека оборачивает методы тестового класса, имена которых соответствуют `patch.TEST_PREFIX`; по умолчанию это `'test'`. То есть запись вроде `@patch.dict("os.environ", {"APP_STAGE": "test"})` над классом `unittest.TestCase` применит одну и ту же временную правку окружения ко всем методам `test_*` внутри класса. Это documented-поведение всех patchers, включая `patch.dict()`. ([Python documentation][1])

Пример:

```python
import os
import unittest
from unittest.mock import patch


@patch.dict("os.environ", {"APP_STAGE": "test"})
class TestStageAwareLogic(unittest.TestCase):
    def test_first_case(self):
        self.assertEqual(os.environ["APP_STAGE"], "test")

    def test_second_case(self):
        self.assertEqual(os.environ["APP_STAGE"], "test")
```

Это удобно, когда у серии тестов один и тот же обязательный фон. Но использовать class decorator стоит аккуратно. Если один тест в классе требует совсем другого env-состояния, общее декорирование начинает скрывать контекст и ухудшает читаемость. В таких местах локальный `with patch.dict(...)` обычно честнее: у подмены есть явная граница, и читатель сразу видит, где именно начинается и заканчивается изменённая среда. Документация даёт class decorator как возможность, а не как рекомендацию использовать его всегда. ([Python documentation][1])

## Где `patch.dict()` помогает, а где уже нет

У `patch.dict()` очень понятная зона ответственности: временно менять содержимое mapping-источника. Поэтому он прекрасно подходит для конфигурации, env, registry и `sys.modules`. Но если код читает значение один раз, кэширует его в глобальную переменную, передаёт дальше как обычный аргумент или строит отдельный словарь для `subprocess.run(env=...)`, то после этого источником истины уже перестаёт быть исходный mapping. В этот момент надо тестировать следующий слой поведения: reload модуля, фабрику настроек, объект конфигурации или формирование `env` для дочернего процесса. Документация по импорту и `subprocess` как раз показывает эти границы: импорт выполняет module-level code, `reload()` переисполняет его, а `subprocess` при явно заданном `env` использует именно переданный mapping. ([Python documentation][5])

Это важный методический момент. Хороший тест на `patch.dict()` не пытается сделать вид, что всё приложение продолжает динамически читать один и тот же словарь на каждом шаге. Он честно отвечает на вопрос: **читает ли код в этой точке именно этот mapping, и читается ли он именно сейчас**. Если да, `patch.dict()` — отличный инструмент. Если нет, значит, подменять уже нужно другую границу. ([Python documentation][1])

## Частые ошибки

Первая ошибка — патчить `os.getenv`, когда код в одном месте использует `getenv`, а в другом — прямой доступ через `os.environ[...]`. В такой ситуации подмена `os.getenv` покрывает только часть сценария, а `patch.dict("os.environ", ...)` покрывает обе ветки, потому что `getenv()` использует `os.environ`. ([Python documentation][2])

Вторая ошибка — тестировать дефолты без `clear=True`. Если код должен использовать значение по умолчанию при **отсутствии** ключа, а Вы просто накладываете пару значений поверх текущего окружения, тест может пройти за счёт чужих переменных, уже лежащих в процессе. `clear=True` нужен именно для герметичного состояния. ([Python documentation][1])

Третья ошибка — ожидать, что `patch.dict("os.environ", ...)` изменит модульную константу, вычисленную при импорте. Не изменит. Если значение было прочитано на module level, его нужно перечитать: обычно через `importlib.reload()` внутри patched-области или через более тестируемый дизайн, где env читается не на импорт, а в явной функции. ([Python documentation][4])

Четвёртая ошибка — использовать `os.putenv()` в тесте как основной способ правки окружения. Документация `os` прямо предупреждает, что вызов `putenv()` напрямую не меняет `os.environ`. Для большинства тестов безопаснее и предсказуемее менять `os.environ` напрямую, а значит — использовать `patch.dict()`. ([Python documentation][2])

Пятая ошибка — забывать о платформенной нормализации имён на Windows. Если Вы пишете кроссплатформенные тесты, используйте верхний регистр в именах env-ключей. Это делает намерение теста однозначным. ([Python documentation][2])

## Заключение

`patch.dict()` — это не вспомогательный экзотический patcher, а основной инструмент для тех тестов, где источник поведения — словарь или словареподобный объект. Его сила в том, что он меняет именно **данные**, по которым принимает решение код под тестом, и затем аккуратно возвращает всё в исходное состояние. Для простого словаря конфигурации это даёт чистые unit-тесты без лишней подмены функций. Для `os.environ` — изоляцию от машины разработчика и CI. Для `sys.modules` — возможность контролировать импорт необязательных зависимостей. Для собственных mapping-контейнеров — тестируемость без переписывания production-кода. ([Python documentation][1])

Главный навык здесь не в том, чтобы запомнить сигнатуру `patch.dict()`, а в том, чтобы видеть реальную границу: где именно код читает mapping, что именно он из него берёт и происходит ли это чтение в момент теста, а не раньше. Если ответ положительный, `patch.dict()` почти всегда оказывается самым прямым и самым честным способом написать проверку. Если чтение уже произошло при импорте, если значение закешировано, если subprocess получил свой отдельный `env`, значит, пора тестировать уже другой слой. И это, пожалуй, главный критерий зрелого теста: он патчит не «что-нибудь, чтобы стало зелёным», а ту точку, которая действительно определяет поведение. ([Python documentation][5])
